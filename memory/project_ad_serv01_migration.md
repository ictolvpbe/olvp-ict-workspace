**Geplande werf (vastgelegd 2026-06-08)** — hoort bij het **leegmaken van VLAN 10** (user-VLAN): servers eruit, tier-discipline herstellen (VLAN 33 `VL-SD-SERVICE` = AD/print/file, zie [[project-infrastructure-params]]).

**Doel**: prod AD-DC + Enterprise CA `SRVV-INFRA002` verhuizen van VLAN 10 naar **`SRVV-SERV-01`** in VLAN 33.
- Huidige AD-DC: `SRVV-INFRA002`, DNS `srvv-infra002.olvp.int` → **10.10.0.10 + 10.10.0.11** (VLAN 10).
- Geverifieerd 2026-06-08: **multi-homed** — LDAPS antwoordt óók op **10.33.0.10** (VLAN 33) met issuer `CN=olvp-SRVV-INFRA002-CA`. Dus INFRA002 heeft al een VLAN-33-poot. "VLAN 10 leegmaken" = de VLAN-10-poot weghalen + AD/CA schoon op SRVV-SERV-01 (10.33.0.10) zetten. **Exacte huidige topologie (1 host multi-homed vs. al een 2e VM op .10) eerst bevestigen** vóór planning.
- Prod-CA = `olvp-SRVV-INFRA002-CA`. CA-naam bevat de oude hostname → illustreert waarom **key-behoudend** migreren moet: naam + key blijven, ook op SERV-01.

**Dry-run = de test-domein-CA-migratie** (besloten 2026-06-08, "optie C"): `olvp-SRVV-TEST-OLVP-CA` van de af te bouwen 2016-DC `SRVV-TEST-OLVP` (10.200.0.9) → `SRVV-TST-WIN-01` (10.200.0.10). Runbook **`platform-handbook/hosting/operations/migrate-adcs-test-ca.md`** (RB-2026-ADCS-TEST). Deze test deblokkeert tegelijk de Keycloak `olvp-dev`-realm ([[project-identity-architecture]], RB-2026-KC-OLVPDEV) én valideert de key-behoudende AD CS-migratieprocedure (Backup-CARoleService + "use existing private key" + zelfde CA-naam + **AIA/CDP-hostname herzien**) vóór we aan de prod-AD komen. Lessen uit de test 1-op-1 toepassen op prod.

**Dry-run in uitvoering (2026-06-08)** — test-CA `olvp-SRVV-TEST-OLVP-CA` op `SRVV-TEST-OLVP` (10.200.0.9): Root-CA (self-signed, DC=test/DC=olvp), **SHA1-thumbprint `32:A1:DA:04:97:83:19:AF:46:A0:3E:83:38:1F:28:81:E8:23:1F:93`** (= referentie die ná migratie identiek moet zijn = key-behoud-bewijs), geldig tot 2071, sha512/RSA-2048, CSP = **Microsoft Software Key Storage Provider** (software-KSP → key exporteerbaar via `Backup-CARoleService`, geen HSM). Templates incl. DomainControllerAuthentication/KerberosAuthentication aanwezig. **AIA/CDP = default LDAP-based config (geverifieerd): hostname-neutraal** — de enige URL's met vlag-bit 2 (ADDTOCERTCDP/AIA, = komt in uitgegeven certs) zijn de `ldap:///CN=%7,…`-paden (op CA-naam, in AD); de `http://%1/…`-hostname-URL's staan op vlag `0:` (niet in certs). → **stap 7 (AIA/CDP-herconfig) is hier een no-op**; voor PROD eerst checken of de prod-CA óók LDAP-based is (zo niet → wél herconfig). **Back-up compleet (2026-06-08)** via `certutil -backupKey` + `-backupDB` (Backup-CARoleService leverde niks op): `C:\CABackup\olvp-SRVV-TEST-OLVP-CA.p12` (key+cert) + `DataBase\*.edb` + `caconfig.reg`. **CA-MIGRATIE GESLAAGD (2026-06-08)**: rol verwijderd op .9 → op .10 PFX geïmporteerd (`Import-PfxCertificate`) + CA geconfigureerd via de **GUI-wizard** ("Use existing private key → Select a certificate", koos `olvp-SRVV-TEST-OLVP-CA`) → DB gerestored → CertSvc up, `certutil -ping` alive, **thumbprint `32A1DA…` matcht = key-behoud bevestigd**. Geleerde lessen (in runbook RB-2026-ADCS-TEST verwerkt): (1) `Install-AdcsCertificationAuthority -CertFile` botst met `-OverwriteExistingKey`/`-CACommonName` → `AmbiguousParameterSet`; GUI-wizard betrouwbaarder dan PowerShell voor key-reuse; (2) `certutil -restoreDB` eist een LEGE DB-map (`C:\Windows\System32\CertLog`) → verse wizard-DB + stale `.jfm` + `archive-*` eerst wegmoven, anders `ERROR_DIR_NOT_EMPTY`; (3) na `Start-Service CertSvc` ~5 s wachten vóór ping (anders 1722). `Backup-CARoleService`/`Restore-CARoleService` werkten niet → `certutil -backupKey/-backupDB/-restoreDB` wel. **Stale CA-restant op .10**: `olvp-SRVV-TST-WIN-01-CA` (CA-poging van 24/06/2025, `.jfm` in CertLog) → verweesde AD-registratie nog op te ruimen (follow-up). **DC-cert + LDAPS op .10 LIVE (2026-06-08)**: auto-enroll stond niet aan (GPO) → handmatig `certreq -enroll -machine "DomainControllerAuthentication"` gaf de cert; `renewServerCertificate` via rootDSE (PowerShell, niet cmd) liet NTDS hem oppikken → `10.200.0.10:636` presenteert cert, issuer `olvp-SRVV-TEST-OLVP-CA`, geldig juni 2027. **DRY-RUN VOLLEDIG GESLAAGD.** **Keycloak olvp-dev re-pointed naar .10 (2026-06-08)**: connectionUrl `ldaps://srvv-tst-win-01.olvp.test:636`, sync OK ("0 imported, 688 updated"), geen PKIX-fout. → ITEM 1 (CA-migratie + re-point) compleet. **Follow-ups**: stale `olvp-SRVV-TST-WIN-01-CA` AD-registratie opruimen; .9 als DC retiren; truststore-cruft op SRVV-ID-01 (`srvv-in fra002.crt` met spatie + `srvv-test-olvp.crt`/.9) opruimen.

**Waarom deze werf bestaat**: opgedoken als prereq van slice C (Keycloak realm olvp-dev → olvp.test LDAPS). De test-DC SRVV-TST-WIN-01 had geen LDAPS-cert; bij het oplossen bleek de olvp.test-CA op een af te bouwen 2016-box te staan. User koppelde dit aan de grotere prod-ambitie (VLAN 10 leegmaken + AD naar VLAN 33).

**Risico/aanpak (prod)**: AD-DC-migratie = kritieke infra → maintenance-window tijdens schoolvakantie ([[feedback-risk-aware-changes]]); CA key-behoudend; AIA/CDP/DNS-continuïteit; multi-homing-poot weghalen in dezelfde maintenance. Diagnoses op de test-CA waren groen (CertSvc up, `certutil -ping` OK, RPC-pad open) → weg A (enkel enrollen) zou werken, maar C gekozen omwille van de dry-run-waarde.

**Status**: planning. Tracker NET-2 (verhuizing) + NET-3 (opruim dry-run) + **NET-4 (DNS-client-cutover, nieuw 2026-06-10)**. Geen deadline; gekoppeld aan de VLAN-10-leegmaak-werf.

## Update 2026-06-10 — topologie bevestigd + DNS-client-cutover (NET-4) gepland

**Topologie verhelderd (was open vraag in regel 12):** `10.33.0.10` = **`SRVV-SERV-01`, een distinct gepromoveerde DC** (géén loutere multi-homing-poot van INFRA002). Bewijs: forward A-record `srvv-serv-01.olvp.int`→`10.33.0.10` + hij staat in de **NS-set van `olvp.int`** (adm001, infra002, bsw001, bsp001, edu001, **serv-01**). AD-geïntegreerde zone repliceert er dus al naartoe → hij kan gezaghebbend antwoorden. INFRA002 zelf = multi-homed `10.10.0.10`+`10.10.0.11`. _(De 2026-06-08-note "INFRA002 antwoordt óók op 10.33.0.10:636" was vermoedelijk al SERV-01 vanaf VLAN 35; reachability is per-source-VLAN.)_

**Knelpunt voor client-cutover:** `10.33.0.10` is **niet bereikbaar op :53 vanaf client-VLAN's** (getest vanaf VLAN 34: ping/UDP53/TCP53 falen, traceroute stopt op de VLAN-34-gateway = firewall default-deny VLAN 34→33). Wél bereikbaar op :636 vanaf VLAN 35 (Keycloak-pad ID-001/002). → vóór we clients kunnen ompunten moet het `:53`-pad open (firewall-rules **D-001**/**D-002** in de matrix).

**NET-4 — gefaseerde DNS-client-cutover (besloten 2026-06-10, vóór NET-2):** clients vooraf naar de vervang-DC laten wijzen ipv pas tijdens de maintenance — reversibel, lager risico, ontkoppelt DNS van de CA/VLAN-poot-verhuizing.
1. **Firewall**: per client-zone (+ VPN-pool) `→ 10.33.0.10:53` UDP+TCP openen (spiegel van de bestaande `→10.10.0.10`-regels) + retourverkeer. Matrix D-001 (VPN) / D-002 (overige).
2. **Verifieer** vanaf een nu-bereikbare client: `dig @10.33.0.10 olvp.int SOA` + een A-record → antwoordt SERV-01 echt gezaghebbend?
3. **Secundair eerst**: in DHCP-scopes (+ VPN-push) `10.33.0.10` als **2e** DNS naast `10.10.0.10` → een periode valideren (geen breuk bij uitval van één).
4. **Primair omzwaaien**: `10.33.0.10` primair maken; `10.10.0.10` als secundair behouden tot INFRA002 effectief gedecommissioned wordt (= deel van NET-2).
5. Vergeet **statische** verwijzingen niet (server-`resolv.conf`, appliance-DNS, eventuele hardcoded `10.10.0.10`).

**VPN-push = push-both** (besloten 2026-06-10): push `10.10.0.10` **én** `10.33.0.10` (zie [[project-hosting-fase1-status]] acc/VPN + `vpn.md` §Split-DNS); de `.33`-helft is afhankelijk van firewall D-001.

**Voortgang 2026-06-10 (user):** firewall **VLAN 34 → `10.33.0.10:53` open** (D-002 deels). **NET-4 stap 2 (verify) = GEDAAN**: `dig @10.33.0.10` bevestigt SERV-01 **gezaghebbend** voor `olvp.int` (SOA `srvv-serv-01.olvp.int`), interne records correct (`myschool-acc`→10.36.0.42) **én** publieke forwarding werkt (google.com/one.com resolven → volwaardige recursive resolver). VPN-config aangepast: **`1.1.1.1` verwijderd** (user, 2026-06-10) → VPN-DNS nu `10.33.0.10` (intern, resolt intern+publiek). Optioneel `10.10.0.10` als secundair voor resilience tot NET-2. VPN-eindtest gebeurt **thuis** (endpoint enkel van buiten bereikbaar) — user koppelt terug. Resterend te verifiëren bij die test: DNS-push komt door + interne naam resolt + acc-UI bereikbaar (firewall **A-011** open?) + step-ca-trust op laptop (cert-warning?).
